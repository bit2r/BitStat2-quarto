GOTCHA.md — BitStat2 작업 시 반복하기 쉬운 실수 (재발 방지)

이 저장소(Quarto + shinylive + webR, iCloud 경로)에서 실제로 겪은 함정과 교훈. 새 작업 전 반드시 읽는다. CLAUDE.md와 함께 로컬 전용(gitignore).

0. 가장 크게 틀렸던 것 — webR 인스턴스가 둘이다 ⚠️⚠️

한 페이지에 webR가 두 개 돌아가며 서로 다른 버전이다. 절대 혼동하지 말 것:

무엇 확장 webR 버전 지정 위치
① {webr-r} 코드셀 — 화면의 “WEBR STATUS 🟢 Ready!” _extensions/coatless/webr (quarto-webr) webr.lua의 baseVersionWebR + webr-worker.js/webr-serviceworker.js의 importScripts
② shinylive 인터랙티브 앱 — {shinylive-r} 블록 _extensions/quarto-ext/shinylive + 자산 자산 버전(_environment의 SHINYLIVE_ASSETS_VERSION) → 그 자산이 특정 webR 번들

교훈: 네트워크/콘솔에서 webr.r-wasm.org/v0.3.3 + repo.r-wasm.org/…/contrib/4.3.3 원격 다운로드가 보이면 대개 ①코드셀이다. ②앱의 속도/버전을 진단할 땐 iframe 안(앱)의 요청과 코드셀 요청을 반드시 분리해서 볼 것. (2026-07-05: 이걸 혼동해 잘 되던 앱 번들링을 성급히 되돌리는 실수를 함.)

1. 버전은 4~5겹이 모두 정렬돼야 한다

quarto-ext/shinylive 확장 · shinylive R 패키지 · 자산(shinylive-X.Y.Z) · webR 런타임 (+ 코드셀은 coatless/webr). - 자산만 올리고(예: 0.10.12) 확장/런타임이 옛 webR면 번들이 무용지물이 되거나 원격 다운로드로 폴백. - webR↔︎R 매핑: 0.3.3→R4.3.3, 0.4.x→4.4, 0.5.x→4.5, 0.6.x→4.6. 패키지는 repo.r-wasm.org/bin/emscripten/contrib/<R버전>/. - webR CDN 접근 가능 버전(2026-07 기준): v0.3.3 ~ v0.6.0 (200). v0.6.1은 403(문서상 최신이나 CDN 미배포) → 실사용 최신은 0.6.0.

2. shinylive 앱 번들링 (속도 핵심)

  • shinylive 자산 0.8.0+ (여기선 0.10.12) 부터 export 시 패키지를 site_libs/…/shinylive-0.10.12/shinylive/webr/packages/*.tgz로 로컬 번들 → 런타임 원격 다운로드 제거.
  • 필요: shinylive R 패키지 0.5.0+ + shinylive::assets_download("0.10.12") + _environment에 SHINYLIVE_ASSETS_VERSION=0.10.12.
  • .Renviron은 안 먹힌다 — shinylive 확장 서브프로세스가 안 읽음. 반드시 _environment.
  • 트레이드오프: 번들로 docs/가 크게 커짐(0.2.3 78M → 0.10.12 ~173M). GitHub 파일당 100MB 제한은 안 넘음.
  • 0.2.3은 “legacy”(번들 없음) — 렌더 로그에 Can't bundle … use 0.8.0 or later 경고.

3. _freeze 가 옛 HTML을 복원한다 (stale)

  • 앱 코드/마크다운을 바꿔도 _freeze가 이전 렌더 HTML을 복원해 변경이 안 보일 수 있다(예: page_sidebar로 바꿨는데 출력엔 fluidPage).
  • 해결: 렌더 전 rm -rf _freeze .quarto docs 후 클린 렌더. 배포 시 _freeze도 새로 미러.

4. 서비스워커 캐시로 옛 앱이 뜬다

  • shinylive 앱은 서비스워커를 등록해 앱을 공격적으로 캐시한다. 재렌더 후에도 브라우저가 옛 앱을 서빙.
  • 검증 시 반드시 새 포트(예: 7801→7809)로 서빙해 SW 스코프를 새로 만들 것. 그래야 최신 앱이 뜬다. (파랑→뮤트 안 바뀌어 보이던 것이 실은 SW 캐시였음.)

5. iCloud 경로가 렌더 산출물을 손상시킨다

  • 저장소가 iCloud Drive에 있어 렌더 중 iCloud가 docs/site_libs를 site_libs 4로 rename하고 index 2.html 등 충돌 중복본을 만든다.
  • 해결: 비-iCloud 로컬 빌드 디렉터리($CLAUDE_JOB_DIR/tmp/…)에서 렌더 → killall bird 후 docs/만 복사 → find … -name '* [0-9]' | xargs rm -rf로 중복 제거. 커밋 직후에도 iCloud가 파일을 지우거나 중복 생성할 수 있으니 재확인.

6. 렌더 엔진 / 파이프라인

  • 모든 콘텐츠 문서에 engine: knitr 필수(front matter). 없으면 shinylive/webr 전용 문서가 jupyter 엔진으로 폴백 → No module named 'yaml' “Quarto Render Error”. _quarto.yml/_metadata.yml의 engine은 자동선택을 못 덮는다.
  • render: allowlist는 pages/**/*.qmd(재귀 glob) 이어야 중첩 모듈이 렌더된다(pages/만으론 건너뜀).

7. 속도 측정

  • shinylive 앱 패키지는 iframe/워커에서 로드 → 메인 페이지 performance API에 안 잡힘. browser_network_requests(iframe 포함) 또는 콘솔로 확인.
  • “WEBR STATUS Ready!”는 ①코드셀 상태다. 앱 속도와 별개. 사용자가 “코드셀은 빠른데 앱은 느리다”고 하면 ②앱을 따로 측정.

8b. 배포는 gh-pages 브랜치로 (2026-07-05 분리)

  • docs/는 이제 main에서 추적 안 함(.gitignore에 /docs/). 빌드 산출물은 gh-pages 브랜치 루트에 있고 GitHub Pages가 거기서 서빙(r2bit.com/BitStat2/). Pages 설정: source={branch: gh-pages, path: /}.

  • 배포 절차(렌더 후, 비-iCloud worktree 사용 — iCloud 파일 손상 회피):

    # 1) 비-iCloud 빌드에서 렌더 → $BUILD/docs (shinylive 0.10.12 · webR 0.6.0)
# 2) gh-pages worktree 체크아웃(비-iCloud 경로) 후 docs 내용 미러
git worktree add "$CLAUDE_JOB_DIR/tmp/ghp" gh-pages
rsync -a --delete --exclude='.git' "$BUILD/docs/" "$CLAUDE_JOB_DIR/tmp/ghp/"
( cd "$CLAUDE_JOB_DIR/tmp/ghp" && git add -A && git commit -q -m "deploy: <요약>" && git push origin gh-pages )
git worktree remove --force "$CLAUDE_JOB_DIR/tmp/ghp"
# 3) 로컬 docs/(gitignore됨)는 미리보기용으로만 유지. main엔 커밋하지 않는다.

  • docs/.nojekyll 필수(site_libs 등 서빙). CNAME은 이 repo에 없음(커스텀 도메인은 org 레벨).

  • main 커밋엔 이제 docs가 안 들어가므로 소스만 가볍게 유지된다.

8c. GitHub Pages 빌드가 “building”에 멈춘다 (source 전환 시)

  • Pages source를 전환하거나 짧은 시간에 여러 push를 하면 빌드가 status: building에 멈춰(duration 0, updated==created) 후속 빌드를 막는다. 사이트는 옛 버전을 계속 서빙.
  • 해결: 새 빌드를 강제 요청 → gh api repos/<owner>/<repo>/pages/builds -X POST. 그러면 현재 source(gh-pages) 커밋으로 새 빌드가 큐잉되어 멈춘 빌드를 밀어낸다. 20~40초 후 built.
  • 확인: gh api .../pages/builds/latest 의 status·commit. commit이 gh-pages 커밋이어야 정상.
  • 완료 후에도 CDN 캐시로 몇 분 지연 가능. curl로 shinylive-0.10.12·v0.6.0·번들 .tgz 200 확인.

8. 일반 규율

  • 성급한 결론/되돌림 금지 — 특히 “속도” 판단 전 ①/② 분리 측정. 되돌리기 전에 진단을 재검증.
  • 대용량 커밋(docs 번들)은 git 이력에 영구 잔존 → 용량 걱정 시 gh-pages 분리/LFS 검토.
  • 정본은 pages/NN/module/index.qmd 인라인 {shinylive-r}. shiny/ 폴더 만들지 말 것.

9. Quarto Live / BitStat2 Live (webR 코드학습 사이트) 함정 — 2026-07-05~06

live/ = Quarto Live(r-wasm/live 0.2.0, webR 0.6.0) 코드학습 사이트. app(shinylive)과 분리.

9a. website × live-html 포맷 충돌

  • 기존 type: website 프로젝트에 format: live-html 페이지를 섞으면 Unable to read the extension 'live' 렌더 실패.
  • 해결: live를 별도 Quarto 프로젝트(live/ 독립 _quarto.yml)로 분리. 그 프로젝트는 _extensions/r-wasm/live를 자체 보유해야 함(상위 _extensions 못 씀).

9b. 엔진 자동선택 → jupyter 실패 (여기도 동일)

  • live 문서에 knitr {r} 셀이 없으면 quarto가 jupyter 선택 → No module named 'yaml'.
  • 해결: 모든 live 문서 front matter에 engine: knitr 필수.

9c. 한글 폰트 — showtext 쓰지 말 것 (canvas 웹폰트)

  • webR canvas에서 showtext는 텍스트를 아예 안 그림(폰트 로드돼도 미표시). font_add_google는 webR 네트워크로 실패.
  • 해결(표준): showtext 미사용.
    1. live/assets/NanumGothic-Regular.ttf + 프로젝트 resources: [assets](→ _site/assets/ 서빙).
    2. live/_header.html에 @font-face{font-family:'NanumGothic';src:url('/assets/NanumGothic-Regular.ttf')...} → format: live-html: include-in-header: _header.html.
    3. R: theme(text=element_text(family="NanumGothic")). webr 패키지 = ggplot2만.
  • 잔여 경고 'NanumGothic' not found in PostScript font database = 무해(실제 렌더는 canvas). suppressWarnings(print(p))로 억제.

9d. ★YAML input:의 짧은 이름이 불리언으로 파싱됨

  • #| input: 목록의 n이 YAML 1.1 불리언(n=no=false)으로 파싱 → data-input="[false,...]" → R에서 n 미주입 → matrix/switch 에러.
  • 해결: input 이름을 따옴표로: - "n". (y/n/yes/no/on/off/true/false 계열 전부 위험)
  • 진단법: 렌더된 HTML의 data-input="..." 확인(제일 확실).

9e. ★OJS↔︎webR 반응형은 input: 말고 define: 패턴을 써라

  • 틀린 패턴(문제 많음): webr 셀 #| input: [n]로 OJS 값 주입 → 9d 버그·edit 문제·초기 미실행.
  • 맞는 패턴(문서 penguins 예제):
    • webr 셀: #| autorun: true + #| edit: false + #| output: false + #| define: [함수명] → R 함수를 정의해 OJS로 export.
    • ojs 셀: viewof 슬라이더들 + 함수명(a, b, c) 직접 호출. 슬라이더 바뀌면 OJS가 R 함수를 재호출(반응형).
  • 주의: define/setup 셀은 autorun: true 없으면 자동 실행 안 됨(→ 함수 미정의 → OJS "함수 is not defined"). edit:false만으론 실행 안 함.
  • 로딩 중 "함수 is not defined" OJS 에러는 webR 부팅 전 일시적(문서에 명시). webR 완료 후 해소.
  • OJS select로 R에 값 넘길 땐 Inputs.select(["a","b"], {format: x=>라벨맵[x]})(plain 문자열+format). new Map(...)은 값 타입 깨짐.

9f. 자동화 브라우저(Claude-in-Chrome)는 webR 출력을 못 봄

  • webR canvas 플롯은 스크린샷에 안 잡히고, 셀 내부가 격리 컨텍스트라 top-document JS(querySelectorAll)로도 canvas/버튼이 0으로 나옴.
  • 텍스트 에러/경고와 렌더된 HTML의 data-* 속성은 읽힌다 → 배선(input/define/edit) 진단은 HTML로, 최종 플롯 시각확인은 사람 브라우저로.
  • 미리보기는 COOP/COEP 헤더 서버 권장(webR 안정): Cross-Origin-Opener-Policy: same-origin + Cross-Origin-Embedder-Policy: credentialless.

9g. ★★반응형(input:/define:)이 이 환경서 런타임 미동작 — 확정 (2026-07-06)

  • 증상: HTML 배선은 완벽(data-autorun="true" data-edit="false" data-define="clt_plot" 또는 data-input=["n",...])한데, 런타임에서 셀이 자동 실행 안 됨 → define 함수가 OJS로 export 안 됨 → 지속적 "clt_plot is not defined". input 패턴도 동일(주입 안 됨/수동 Run만).
  • 환경: Quarto Live 0.2.0 + Quarto 1.10.11(확장 quarto-required ≥1.4, 구버전 대상 테스트 추정). 문서 penguins 예제 그대로 써도 자동실행 안 됨.
  • 즉 이 환경선 슬라이더→플롯 자동 반응이 불가. 배선/문법 문제 아님(다 맞음). 런타임/버전 비호환.
  • 당장 되는 대안(권장): 편집형 셀(학습자가 n <- 30 고쳐 “Run Code”). 이건 확실히 동작(한글 플롯 렌더 확인됨, 커밋 a87e5f0 방식). 교육적으로도 무방.
  • 반응형 살리려면(후속): Quarto Live 버전 올리기/내리기 실험, Quarto 버전 조정, 또는 r-wasm/quarto-live 이슈 확인. 우선순위 낮춤.
  • 교훈: webR 반응형은 배선(HTML data-*)이 맞아도 런타임이 별개. 배선 확인 후 반드시 실브라우저 런타임 확인(자동화 브라우저는 canvas 못 봄).
